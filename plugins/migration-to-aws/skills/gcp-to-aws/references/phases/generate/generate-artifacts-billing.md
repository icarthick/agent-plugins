# Generate Phase: Billing Skeleton Artifact Generation

> Loaded by generate.md when generation-billing.json and aws-design-billing.json exist.

**Execute ALL steps in order. Do not skip or optimize.**

## Overview

Generate **skeleton Terraform** with TODO markers for billing-only migrations. These configurations are NOT deployable as-is — they require manual configuration refinement based on actual infrastructure discovery.

Every resource block includes TODO markers indicating what information is missing and where to obtain it. The skeleton serves as a starting point that accelerates the migration once IaC discovery or manual audit provides the missing details.

## Prerequisites

Read the following artifacts from `$MIGRATION_DIR/`:

- `aws-design-billing.json` (REQUIRED) — Billing-based service mapping from Phase 3
- `generation-billing.json` (REQUIRED) — Conservative migration plan from Stage 1

If any required file is missing: **STOP**. Output: "Missing required artifact: [filename]. Complete the prior phase that produces it."

## Output Structure

```
$MIGRATION_DIR/
└── terraform/
    ├── main.tf          # Provider config (only if infra track didn't generate it)
    ├── variables.tf     # Inferred variables (only if infra track didn't generate it)
    └── skeleton.tf      # Resource stubs with TODO markers
```

**Important**: If the infrastructure track already generated a `terraform/` directory (because `generation-infra.json` also exists), do NOT overwrite `main.tf` or `variables.tf`. Only generate `skeleton.tf` for the additional billing-only services.

## Step 0: Read Design Inputs

From `aws-design-billing.json`, extract:

1. **Mapped services** (`services[]`) — Services with AWS targets assigned
2. **Unmapped services** (`unknowns[]`) — Services that could not be mapped from billing data
3. **Total service count** from `metadata.total_services`
4. **Confidence note** from `metadata.confidence_note`

From `generation-billing.json`, extract:

1. **Migration timeline** for header comments
2. **Risk level** for confidence tags

## Step 1: Generate main.tf (if not present)

```hcl
  required_providers {
    aws = {
```

## Step 2: Generate variables.tf (if not present)

Only generate if `$MIGRATION_DIR/terraform/variables.tf` does not already exist.

For each mapped service in `aws-design-billing.json`, create variables with:

- **Inferred values**: Variables with known defaults get a `# Verify` comment
- **Unknown values**: Variables without defaults get a `# TODO` comment and a placeholder

```hcl
# =============================================================================
# VARIABLES — BILLING-ONLY SKELETON
# Many values are placeholders. Search for "TODO" to find values that need
# manual configuration.
# =============================================================================

variable "aws_region" {
  description = "AWS region for deployment"
  type        = string
  default     = "us-east-1"  # Verify: From preferences.json
}

variable "project_name" {
  description = "Project name for resource naming"
  type        = string
  default     = "gcp-migration"  # TODO: Set to your project name
}

variable "environment" {
  description = "Deployment environment"
  type        = string
  default     = "dev"
}

# --- Compute Variables ---
# TODO: These values are unknown from billing data alone.
# Check your GCP Cloud Run / GCE / GKE configurations for actual values.

variable "fargate_cpu" {
  description = "Fargate task CPU units"
  type        = number
  default     = 256  # TODO: Check actual CPU allocation in GCP
}

variable "fargate_memory" {
  description = "Fargate task memory in MB"
  type        = number
  default     = 512  # TODO: Check actual memory allocation in GCP
}

# --- Database Variables ---
# TODO: These values are unknown from billing data alone.
# Check your GCP Cloud SQL configuration for actual values.

variable "db_instance_class" {
  description = "RDS instance class"
  type        = string
  default     = "db.t4g.micro"  # TODO: Check actual Cloud SQL tier
```

Generate additional variables for each service in `aws-design-billing.json.services[]`, using billing SKU hints to inform reasonable defaults where possible.

## Step 3: Generate skeleton.tf

For each mapped service in `aws-design-billing.json.services[]`, generate a resource stub.

### Resource Stub Format

Every resource block follows this pattern:

```hcl
# -----------------------------------------------------------------------------
# [AWS Service] — Migrated from [GCP Service]
# Confidence: billing_inferred
# Monthly GCP Cost: $[amount]
# SKU Hints: [sku descriptions]
# TODO: Verify all configuration values against actual GCP resource settings
# -----------------------------------------------------------------------------
resource "aws_[resource_type]" "[name]" {
  # TODO: Verify [attribute] — value inferred from billing data
  [attribute] = [value]

  # TODO: Set [attribute] — not available from billing data
  # [attribute] = "unknown"

  tags = {
    Name       = "${var.project_name}-[name]"
    GCPSource  = "[gcp_service]"
    Confidence = "billing-inferred"
  }
}
```

```hcl
  }
```

## Step 4: Self-Check

After generating all files, verify:

1. **Every mapped service** in `aws-design-billing.json.services[]` has a resource stub in `skeleton.tf`
2. **Every resource** has at least one `# TODO` comment indicating what needs verification
3. **No resource is presented as production-ready** — all have confidence markers
4. **Skeleton warning header** is present at the top of `main.tf` and `skeleton.tf`
5. **No hardcoded credentials** in any file
6. **Tags include `Confidence = "billing-inferred"`** on every resource
7. **Unmapped services** from `aws-design-billing.json.unknowns[]` are listed at the bottom
8. **Did not overwrite** existing `main.tf` or `variables.tf` if infrastructure track already ran
